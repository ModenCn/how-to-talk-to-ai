# 关于 vibe coding 的思考

> 写作草稿 · 2026-06-14（持续更新中）

---

## ①

**原话：** vibe coding 的原理是，这个世界上计算机存在的可能性，vibe coding 是通过 AI 使其快速收敛到某一种具体的可能性。

**扩写：**

设想一个抽象空间：这个世界上所有可能存在的计算机程序，构成一个近乎无限的「可能性空间」。其中每一个点，都是一种具体的软件——可能是能跑的应用，也可能是一堆永远无法编译的乱码。绝大多数点是无意义的；只有极小一部分，对应着真正有用、能解决某个问题的程序。

传统编程，是人用双手在这个空间里**一步一步走**：写一行代码，就是朝某个方向挪一小步；调试、重构，是不断修正方向。程序员凭经验和逻辑，亲手把系统从「什么都不是」一点点搭到「那个具体的可能性」。这个过程慢、依赖技艺，且高度依赖人对路径的掌控。

vibe coding 改变的，是**抵达的方式**。你不再一步步走，而是向 AI 描述你想要的「感觉」（vibe）——目标、约束、风格、意图。AI 凭借它从海量代码中学到的先验，把这束模糊的意图当作一股引力，**几乎瞬间把整个可能性空间收敛**到其中某一个具体实现上。描述越清晰，引力越强，落点越准。

一句话概括它的原理：

> 它不是教 AI 写代码，而是用 AI 把「所有计算机可能的样子」，快速坍缩成「你此刻想要的那一个样子」。

**补充 · 与「无限猴子定理」的对照：**

无限猴子定理：猴子在打字机上随机无限地敲，几乎必然（概率为 1）敲出任意给定文本（如莎士比亚全集）——原理是所有字符序列构成一个空间，随机过程只要时间无限，就会遍历到其中任何一个有限序列；但那个概率小到天文数字，需要的时间远超宇宙年龄。

它正好是 vibe coding 的反面对照——同一个「可能性空间」，两种抵达方式：

| | 猴子写书 | vibe coding |
|---|---|---|
| 同一个 | 可能性空间 | 可能性空间 |
| 怎么找 | 纯随机穷举 | AI 定向收敛 |
| 靠什么 | 运气 | 先验 + 你的意图（引力） |
| 代价 | 无限时间 | 几乎瞬时 |

一句话：AI 干的事，就是把猴子写书所需的「无限时间」，压缩成一句 prompt。

# 第一章：人在 vibe coding 中的必要性

## 为什么人工智能不能替代人类

**核心论点：** 人工智能不能根本性地替代人类，最根本的原因，和量子力学需要「观测者」的原因类似。

**论述：** 量子力学中假想的「观测者」，必须由一个主体去触发。即便这个主体是非意志的，也必须有一个有意志的主体去主动触发它；即便存在「非意志主体间接触发观测」的情况，那也只是可能性自身导致的必然——也就是说，如果一个意志主体使用了任何间接手段、导致了这次量子坍缩，追其根源，仍是意志主体干涉了可能性，使可能性落在了某个固定的区间。

从这个角度讲，**人是可以选择未来的**。

我把这个动作称为「意志坍缩」——意志介入因果，让无数可能里的某一个，落地成现实。

**推论：** 所以，人的意志（或者说主观），是一种具有极其强大力量的事物——因为它可以决定现实事物落下的方向。

**限定（张力）：** 但人也受到「光锥」的约束——可能性是有范围的，人无法跳到这个可能性范围之外。而且这些可能性会如同水往低处流一样，自然地朝某一个方向落下；人的意志是在**对抗**这个自然下落的过程，所以必然会受到阻力。

**点睛：** 这，就是命运的形态。

**回到 vibe coding（结论）：** 所以，vibe coding 这件事物本身，是无法离开人的——它的每一次收敛，本质上都是一次「意志坍缩」。

# 第二章：vibe coding 的本质

**开篇：** vibe coding 的发展是随着时代进步的——也就是说，随着 AI 能力的不断变强，vibe coding 的形态也在不断发生变化。

**思想实验（极限情况）：** 假想一种极限情况——你的任何想法，都能 100% 复刻到「程序的可能性形态」中。

也就是说：**程序 ＝ 你的想法的完整体**。

**现实的偏差：** 但现实中，你和 AI 的沟通只是在进行「数据交换」——AI 无法直接把你脑中的结果复刻出来，所以必然会产生偏差。

**更深一层（源头也模糊）：** 而且同样地，你脑中的结果本身也会产生偏差——因为人脑这套神经网络本身不是确定性产物，它会「模糊」，就像 AI 生成的图片里总有某处是模糊的点一样。

**抽象的双面：** 大脑的「抽象机制」提供了高效的运算能力；但与此同时，它需要一个东西去不断地校准它。

**校准物 ＝ 现实世界（参考系）：** 人的大脑之所以无法完全离开「以现实世界为主体」，根本原因就在于——神经网络需要一个确切的参考标准（参考系）去校准自己。

**参考系必须稳定：** 而这个「标」不能浮动。一旦参考标准本身产生了浮动，神经网络就会失去锚定，陷入一种无序。

**对自递归系统尤为致命：** 尤其是对于一种可以「自递归」的系统来说，这种参考标的浮动，是致命的。

**真智能 vs 模拟智能（判据）：** 无论产生多强大的 AI，只要它需要一个「根因」——需要一个人去推动它行为的第一步、它自己不会产生动因——那就不能视为这个 AI 存在某种真实的智能，只能说它「模拟」了智能。

**正确形态（结论）：** 所以，vibe coding 真正的形态，应该是——以「具体的目标」为导向，AI 与人之间形成「耦合性协同」。

**人在耦合中的根本作用：** 而人之所以必须参与这个耦合步骤，根本原因就是——为了消除「结果」与「目标」之间的偏差。

**本质：** 这，就是 vibe coding 的本质——不断消除「结果」与「目标」之间的偏差。

# 第三章：为什么你不应该看代码

## 一、复杂系统的耦合，与设计导向

**论点：** 从「复杂系统的耦合」和「设计导向」的角度，为什么不应该看具体的代码实现。

下面通过一个简单的思考假设来论证：

**假设设定：**
- **A 方** ＝ 项目的发起方
- **B 方** ＝ 项目的执行方

**场景：** 你（A 方）是项目需求的根源提供者，花了很多钱，找了项目实现方 B 方。

**核心：** 你（A 方）不需要知道任何代码实现——你只知道「你想要什么」；其中所有的具体实现，都由 B 方来完成。

**过程：** 然后你们通过多次的交流，把项目做成你想要的具体交付形态。

**提炼：** 这个过程中，A 方无需看任何一行具体的代码实现。也就是说——A 与 B 之间，存在某种「抽象机制」。

**推论：** 而这层抽象机制，是可以通过神经网络模拟的——所以 AI 必定会替代这中间的抽象层。

**关键概念：** 这里重要的概念是——如果你看了代码，实际上就跳出了你作为 A 方的身份。

**三层模型：** 现在的逻辑是——**A —〔抽象层 B〕— 代码 C**：
- **A** ＝ 你（发起方）
- **B** ＝ 抽象层，也就是前面的 B 方（AI 执行方）——执行方承载的，正是这一层抽象
- **C** ＝ 代码（最终实现）

你绕过抽象层（B）去看了代码（C），是不利于你用「完全抽象化的形式」去使用 AI 的——因为你需要的，是**抽象化沟通的能力**。

**方法论：** 你不应该以任何形式去直接观看代码（C）。你应该去观看**效果（D）**，然后和**抽象层 B**进行沟通。

**现实佐证：** 事实上，在现实中甲方也不会去观看代码。

**收束（接回「耦合」）：** 所以，不应该看代码的原因，正是你需要「耦合」的根本原因——你需要根据**结果**去耦合**目标**。如果你直接去看实现层，这就相当于替代了中间抽象层的工作。

# 第四章：谁对代码负责

**开篇（现实担忧）：** 很多时候你会担心，AI 写的代码落到实际项目上会出问题。

**事实：** 而事实上，不仅 AI 写的代码会出问题，人写的代码也一样会出问题。

**关键论点：** 一个成熟的软件开发公司，是通过「流程」来解决这些潜在问题的，而不是通过某个确切的人。

**强调：** 人会出错，AI 也会出错——这没有任何问题。

**现实：** 项目的甲方需要一个完全不会错的程序，但这在现实中是不存在的。

**真正可行的形式：** 现实中真正存在的形式，是「降低出错后产生的后果的严重程度」。

**核心：** 我们知道这个系统会出错——但我们知道它出错后「最大的严重程度」是什么。

**结论（三道防线）：** 测试流程、沙箱环境、备份冗余——这三者才是保证项目上线的根本，而不是一个从不出错的 AI。

# 第五章：传统编程为什么会被淘汰

**开篇（核心）：** 效率，是一切的关键。

**论述（效率 → 生产力 → 价值）：** 我们身处一个高效的社会——效率决定了生产力，生产力决定了价值。

**论断：** 代码本身不会创造任何价值。代码本身就是代码——代码就是石头。

**价值的来源：** 代码只有被使用了、参与到人的行为中了，才产生价值。

**代码 vs 艺术（等效相反）：** 代码和艺术几乎是「等效相反」的代名词——艺术本身就存在价值；而代码只有「高效的实现」才存在价值，无效代码的价值约等于 0。

**传统编程的本质：** 所以，所有传统编程的手段、方法、类型、抽象对象——这些操作，只是为了让人去（更好地）替代中间的那个抽象层。而实际上，机器只需要执行某个确切的指令。

**高潮推论：** 既然人可以替代中间的抽象层，就说明神经网络可以替代抽象层。所以 AI 不需要中间的过程——AI 可以直接把你的需求，坍缩到某个系统确定的状态。这是注定在未来某一天会发生的。

**当下为何仍需代码：** 而目前之所以还需要编程语言和代码，本质是——人需要对 AI 进行审计，检查、审核 AI。

**类比（安全阀门）：** 这就像所有的安全生产，都会有「自动化」和「手动化」两个开关一样——手动，是在任何关键时刻留的最后一把安全阀门。

**推进：** 而随着时代的变化，AI 审计的效率将远高于人类——所以即便是审计、测试、验证这种工作，也应该交由 AI 执行。

（这里有个隐患：「AI 审计 AI」会撞上第二章「自递归系统对参考浮动致命」——如果执行和审计都是同一个 AI，谁来提供稳定的外部参考？这个隐患的化解，见第六章。）

# 第六章：AI 的局限与人的多样性

（标题暂拟）

**问题（AI 固定模因 vs 人的多样性）：** 但是现实中存在一个问题——AI 存在某种「固定模因」，而人是多种多样的。

**论点：** AI 的思考，无论多全面，都不能涵盖所有的情况。

（这里说的审计，是技术审计，而不是结果审计。）

AI 是有自己的性格的，每一个 AI 模型的性格、特点均有所不同。

AI 作为工具训练的时候，就倾向于输出同一结果。如果每次都输出不同结果——比如我说「苹果是红色的」，它每次都说不一样的颜色——这样就代表这个系统是一个混沌系统。

所以，和它的混沌性相反，同一个 AI，在同一个事物上，有着固有的偏向。

不同的 AI 的性格是不同的。

所以，同一个项目，应该由不同性格的 AI 模型，反复进行验证，以降低出错的概率。

# 第七章：C++ 之死

C++ 以复杂的接口类型和模板类，来降低真实编程中开发的难度、提高系统效率。

而 C++ 是为了解决人在协作中遇到的问题。AI 不需要通过这种手段解决问题——也就是说，AI 可以通过完全的「C 化」，来实现 C++ 中的一切问题。

而绝大多数复杂项目，为了确保项目的可用性，均通过抽象机制来实现，进而避免错误产生实质性的危害。也就是说，这种「使用底层语言去做复杂实现、用开发的低效率换取运行的高效率」本身，已经毫无意义了。

所以，C++ 这种语言，注定会陨灭在历史的长河中。而真正接下来会成为主流的语言，是由三者共同决定的：极高的适配性、极强的通用性，和 AI 的低出错性。

# 第八章：AI 系统的闭环

AI 执行过程中存在的任何开环行为，都应该转换成闭环行为。

因为开环行为阻止了 AI 在现有能力上修复问题的能力，而闭环则强化了它的能力，让它有参考基准点，可以去修改。

所以在开发流程中，应该把开环系统转变成闭环系统，给 AI 一个反馈。

这也是不应该绕过抽象层去修改代码的另一个原因——你改了，AI 不知道，这间接地影响了 AI 在后续的执行效率。

---

# 第九章：注意力机制，和笔记

在能力不强的模型中，注意力需要手动维持。而笔记，是提高 AI 在某些事物上的注意力的方法。

在能力更强的模型中，笔记和注意力维持，能确保它在项目中实现的一致性。

# 第十章：提示词污染

如果一个模型足够强大，你应该尽量地少描述——你的描述会限制 AI 发挥自身能力。

意思是：你和 AI，总有一个人在这件事情上理解得更透彻。但如果更透彻的那个是 AI，你用你的理解描述给它，反而会降低它用更好的方法去实现你目标的可能性。

所以，越弱的 AI，你描述得越应该具体，防止它按照自己的错误继续错下去（诱导结果坍缩到正确方向）；而越强的 AI，你越应该让它自己落到更优的方向去——你说得越多，越有可能把它引导到次优解上。
